# 高级 AI 测试工程师 / Lead · 模拟面试 Q&A

> 基于候选人背景：**企业级 AI（RAG & Agent）测试专家**，擅长检索指标评测、多模态文档处理、向量检索质量、Kubernetes 性能稳定性测试、AI 治理与可观测性。

---

## 第一部分：RAG 系统评测

---

### Q1：你们的 RAG 系统是怎么评测的？只说准确率和召回率吗？

**✅ 回答：**

不会只说准确率和召回率。RAG 评测必须拆成**三层**：

**第一层：检索评测**
- Context Precision（上下文精确率）—— 召回的内容里有多少是真有用的
- Context Recall（上下文召回率）—— 必须用的信息有没有全部召回
- 核心问题：**准不准，全不全**

**第二层：生成评测**
- Faithfulness（忠实度）—— 答案有没有自己脑补
- Answer Relevancy（答案相关性）—— 有没有答到点上

**第三层：业务评测**
- 用户满意度、人工抽检通过率、Bad Case 率、转人工率、追问率

> 💡 **指标反推定位问题**：Faithfulness 高但 Relevancy 低 → 检索上下文方向不对；Context Recall 高但 Faithfulness 低 → 模型在乱编。

---

### Q2：RAG 上线前怎么准备测试用例？

**✅ 回答：**

准备**黄金问题集（Golden Test Cases）**，来源三个：
- 文档内容、高频用户问题、高风险业务问题

每条包含三要素：**用户问题 + 标准答案 + 对应的正确 Chunk**

> 后续每次改文档、改 Prompt，都必须重新跑一遍。

---

### Q3：怎么保证 AI 生成的测试用例是可用的？

**✅ 回答：**

最大的风险是**看起来合理但原文根本没有依据**。

四步走：
1. **原文锚定**：每条用例必须反向找到原文 Chunk，验证 Chunk 里真有这个答案
2. **分层抽检**：高风险用例 100% 人工审，中低风险分层抽检
3. **LLM Judge 辅助验证**：把（问题 + Chunk + 答案）三者一起丢给大模型判断
4. **人工兜底**：LLM Judge 不能完全依赖，人工验证才是最后安全阀

---

### Q4：如何用指标定位 RAG 系统的问题？

**✅ 回答：**

| 指标现象 | 问题定位 | 可能原因 |
|---------|---------|---------|
| Faithfulness 高 + Relevancy 低 | 检索召回的上下文方向不对 | 召回内容与问题不匹配 |
| Context Recall 高 + Faithfulness 低 | 资料找到了但模型在乱编 | Prompt 约束不够 / 模型能力问题 |
| Context Precision 低 + Recall 低 | 别怪大模型 | Embedding 模型 / Chunk 切分策略 / Re-rank 有问题 |

## 第二部分：RAG 文档处理

---

### Q5：RAG 处理 Word 和 PDF 的核心难点是什么？

**✅ 回答：**

> **难点根本不在检索，在最前面那一关——解析文档。**

| 类型 | 难点 |
|------|------|
| **Word** | 嵌入的图片、公式、文本框、Excel 对象抽不干净就丢信息 |
| **PDF** | 版式格式没有语义结构，双栏排版交叉读出来是乱码，表格跨页行列全乱，扫描件没 OCR 就是一个字都拿不到 |

**工程解法**：
- 版面解析（StructureLLaMA / PaddleLayout）识别标题、表格、阅读顺序
- 复杂表格 + 扫描件：多模态大模型，把页面当图片看
- 置信度兜底：宁可少召回，也不要召回错的

---

### Q6：切片怎么切才正确？为什么不能粗暴按字数切？

**✅ 回答：**

**最大问题**：把一句话、一张表、一个完整逻辑从中间砍断，召回的片段缺头少尾，模型自然答错。

**正确做法：结构感知切片**
- 顺着标题层级和段落边界切
- 表格和代码块**整块保留**，不要拆散
- 常见配置：400~800 token，重叠 20%

> 💡 研究数据：结构感知的切片在专业领域能把准确率提升 **40% 以上**。

---

### Q7：RAG 检索阶段怎么做？

**✅ 回答：**

**多路召回**（不要只靠向量检索）：
- 向量检索：找语义相近的
- 关键词检索：找精确命中的术语、编号
- 两路合并后过 **Re-rank**，把最相关的顶上去

组装上下文时**必须带上来源引用**，告诉用户答案出自哪个文件、哪一页。

## 第三部分：Agent 评测

---

### Q8：为什么评估一个 Agent 比传统软件测试难得多？

**✅ 回答：**

| 对比维度 | 传统测试 | Agent |
|---------|---------|-------|
| 输出确定性 | 强 | 弱，LLM 是概率模型 |
| 评估标准 | 客观明确 | 主观模糊 |
| 自动化难度 | 低 | 高 |
| **核心目标** | 证明没有问题 | **感知退化，在性能变差时第一时间发现** |

> 💡 **关键思维转变：从"证明正确"到"感知退化"。**

---

### Q9：Agent 评估的核心指标有哪些？

**✅ 回答：**

| 指标 | 看什么 |
|------|--------|
| **准确率** | 答案对不对 |
| **工具调用成功率** | Agent 使用工具的能力（包括：该调时调、不该调时不调、参数正确） |
| **任务完成率** | 最贴近用户实际感受 |
| **平均步骤数** | 效率，步骤越少 = 执行越快、成本越低 |

---

### Q10：什么是 LLM Judge？为什么有效？

**✅ 回答：**

**LLM Judge** = 用强模型（如 GPT-4）评估另一个模型的输出。

**为什么有效**：
- 判断力接近人类专家，能理解细微之处、识别事实错误
- 成本远低于大规模人工审核

通常从三个维度打分：**正确性 + 完整性 + 相关性**。

---

### Q11：如何构建自己业务的 Agent 评估体系？

**✅ 回答：**

四步走：
1. **构建测试集**：50~100 条，覆盖主要场景
2. **LLM Judge**：平衡成本与精度
3. **胜率对比**：新版本 vs 基线版本，判断是否更好
4. **人工验证兜底**：关键场景必须保留人工审核

---

### Q12：Agent 评估如何工程化落地？

**✅ 回答：**

- 代码提交后 **CI/CD 自动触发**评估任务
- 用 LLM Judge 打分 + 计算核心指标
- 指标相比基线下降超过阈值（如准确率跌 5%）→ **自动阻止合并 + 通知负责人**
- 生产环境：**每日抽样真实对话**进行评估，发现异常立即告警

## 第四部分：Agent 记忆管理

---

### Q13：Agent 的记忆管理怎么做？只用向量数据库 + RAG 够吗？

**✅ 回答：**

> 只用这一招说明你的 Agent 停留在 **DEMO 阶段**。

向量数据库有两个致命局限：
1. **精确查找弱**：擅长模糊匹配，但"用户的手机号是多少"这种需要精准查找
2. **时间盲区**：用户上周在上海，这周搬北京了——两条相似记录同时被捞出来，分不清哪条是当前状态

**生产级分层设计（四层）**：

| 层级 | 内容 | 读写策略 |
|------|------|---------|
| 会话原数据 | 本次会话基本信息 | 用完不持久化 |
| 用户结构化档案卡 | JSON 格式：偏好、身份、历史决策 | **精准键值查找**，不走向量检索 |
| 近期对话摘要 | 最近几次会话的主题和关键词 | 保留语义，压缩 token |
| 滑动窗口 | 当前上下文 | 超出 token 上限丢弃最老的信息 |

---

### Q14：Agent 记忆的时间状态问题怎么解决？

**✅ 回答：**

**双时间机制**：

| 时间戳 | 说明 |
|--------|------|
| **Valid Time** | 信息在现实世界里的有效时间段 |
| **Transaction Time** | 信息是什么时候被写入系统的 |

两个时间戳配合，才能正确处理状态变更（如用户搬家）。

**核心原则**：状态变更要**覆写**，不是让新旧信息在向量空间里互相污染。

---

### Q15：什么是程序性记忆？

**✅ 回答：**

把 Agent 多次成功完成某类任务的交互轨迹，固化成可复用的 **Skill**，下次遇到同类任务直接调用，而不是从零规划。

```
多次成功经验 → 固化成 Skill → 下次同类任务直接调用
```

让 Agent 具备**越用越聪明**的能力。

## 第五部分：多模态文档处理（结合候选人背景）

---

### Q16：OCR 和多模态处理在 RAG 中怎么用？

**✅ 回答：**

| 场景 | 方案 |
|------|------|
| 扫描件 PDF | OCR 识别文字（Tika / PaddleOCR） |
| 复杂表格跨页 | 多模态大模型，把页面当图片理解 |
| 版面结构（双栏、标题） | StructureLLaMA / PaddleLayout 识别阅读顺序 |

**置信度兜底**：解析阶段做置信度判断，识别不出来的不要硬塞，该走人工或多模态兜底就兜底。

## 第六部分：技术基础设施（K8s + 性能）

---

### Q17：K8s 环境下怎么做 AI 系统的性能和稳定性测试？

**✅ 回答：**

| 测试类型 | 关键点 |
|---------|--------|
| **性能测试** | P99 延迟、吞吐量、RT 分布 |
| **稳定性测试** | 长时间运行，监控内存泄漏、指标漂移 |
| **容灾恢复** | 节点故障时服务是否自动恢复，SLA 是否达标 |
| **资源压测** | 找系统在极限负载下的瓶颈 |

重点关注：**SLA 达成率 + P99 延迟 + 故障恢复时间**。

## 第七部分：AI 治理与可观测性

---

### Q18：Agent 怎么建立可观测性框架？

**✅ 回答：**

**Trace 记录三层链路**：

| 层级 | 记录内容 | 关键指标 |
|------|---------|---------|
| **检索层** | 用户原始问题、召回 Chunk、Re-rank 排序变化 | Hit Rate@K、无结果比例、低相似度比例 |
| **上下文层** | 哪些 Chunk 被放进 Prompt、哪些是背景 | 核心 Chunk 覆盖率、噪音比率 |
| **生成层** | 最终回答、是否引用了 Chunk | 忠实度、幻觉率、答案相关性 |

**生产监控**：自动化 + 人工抽检混合，每天抽样高流量 / 高点踩 / 追问率高的问题人工复核。

---

### Q19：什么是 IG 漂移？怎么监控？

**✅ 回答：**

**IG 漂移**：系统在检索、重排、生成过程中偏离了用户问题和资料，答非所问。

**为什么会漂移**：
- 检索召回了一堆无关内容
- Re-rank 把错误内容排到了前面
- 大模型自己脑补了没有的内容

**监控核心**：记录完整 Trace，用指标（无结果率、低相似度率、幻觉率）发现漂移。

## 第八部分：综合系统设计

---

### Q20：设计一个企业级 RAG 系统的测试体系，你会怎么做？

**✅ 回答：**

**上线前（三层评测）**：
- 组件级：单独测 Embedding / Re-rank / Chunk 策略（Hit Rate、MRR、NDCG）
- 端到端：完整链路（Context Precision/Recall、Faithfulness、Answer Relevancy）
- 黄金用例：50~100 条，覆盖主要场景 + 高风险场景

**上线后（四层监控）**：
- 检索层：召回率、无结果率、低相似度率
- 上下文层：核心 Chunk 覆盖率、噪音比
- 生成层：忠实度、幻觉率、答案相关性
- 用户反馈层：点赞率、追问率、转人工率

**工具选型**：RAGAS（快速迭代）、Trulens（全链路追踪）、CI/CD 自动触发 + 安全阈值阻断

**兜底**：每日人工抽检 200 条高价值问题。

## 一句话总结

```
RAG评测：检索看准全、生成看编答、定位靠指标反推
Agent评测：四层分层设计 + LLM Judge + CI/CD自动化 + 感知退化不是证明正确
记忆管理：四层分层 + 双时间机制 + 程序性Skill固化
文档处理：版面解析 + 结构感知切片 + 多模态/OCR兜底
工程落地：K8s性能压测 + SLA/P99监控 + 每日人工抽检
```
