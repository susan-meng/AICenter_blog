### 1. 你负责的 RAG 全链路检索质量评测，具体用了哪些评测指标？

**参考答案：**

RAG 检索质量评测围绕四个核心维度展开：

**① 召回率（Recall）**
- 构建标准评测集（Query → 期望召回文档），量化每个 Query 的 Top-K 召回是否包含正确答案
- 使用命中率（Hit Rate@K）作为主要指标，K 通常取 3、5、10

**② 排序质量（Ranking Quality）**
- 使用 NDCG（归一化折损累计增益）评估排序效果，确保相关文档排在前列
- 结合 MRR（平均倒数排名）衡量整体排序合理性

**③ 语义相关性（Semantic Relevance）**
- 通过人工标注或 LLM 评估生成结果与 Query 的语义匹配度
- 验证向量化模型对同义词、多义词的处理能力

**④ 上下文连贯性（Context Coherence）**
- 验证切片（Chunk）边界是否破坏语义完整性
- 检查元数据提取的准确性，包括文档标题、段落编号等

在 AnyShare 智能助手项目中，覆盖文本、表格、图片、视频多模态文档场景，结合 Apache Tika + OCR 全链路解析，确保向量化结果符合业务预期。

### 2. 召回率、排序质量的量化标准是什么？如何验证向量化和切片语义连贯性？

**参考答案：**

**量化标准：**

| 指标 | 计算方式 | 合格阈值（参考） |
|------|---------|----------------|
| Recall@K | 召回正确文档数 / 总正确文档数 | ≥ 85% |
| NDCG@K | 排序位置加权相关性求和 | ≥ 0.75 |
| MRR | 正确文档首次出现位置的倒数均值 | ≥ 0.70 |

**向量化和切片验证方法：**

1. **向量化验证**：
   - 使用余弦相似度计算 Query 与候选文档的语义距离，阈值通常设置在 0.75 以上
   - 对比不同 Embedding 模型（如 OpenAI text-embedding-3-large、本地模型）的召回差异
   - 通过降维可视化（t-SNE/UMAP）检查向量化空间分布是否合理

2. **切片语义连贯性验证**：
   - 设计语义连贯性评分用例：连续切片应能还原完整语义段落
   - 人工抽检切片边界，验证是否存在一句话被截断在两个 Chunk 中
   - 使用 LLM 对切片内容进行摘要生成，检验语义损失率

### 3. 请详细描述 AI Agent 效果评测的测试方法，涵盖哪些维度？

**参考答案：**

AI Agent 效果评测覆盖以下核心维度：

**① 语义理解准确性**
- 设计多样化 Query 测试集，涵盖单轮问答、多轮对话、模糊意图等场景
- 验证 Agent 对用户意图的理解准确率，区分有效响应与无效响应
- 评估上下文窗口管理能力（长对话截断、关键信息保留）

**② 工具调用链路可观测性**
- 验证 Function Calling / MCP 机制的工具调用是否正确触发
- 通过 Dolphin Runtime 轨迹数据追踪完整执行链路
- 检查工具调用参数传递的准确性和异常处理

**③ 决策可解释性**
- 验证 Agent 决策路径是否可追溯，输出结果是否有对应推理依据
- 设计反事实测试用例，检验决策逻辑的一致性

**④ 安全合规与策略审计**
- 验证权限精确控制：不同角色用户是否只能访问授权范围的数据
- 内容安全过滤：敏感信息是否正确过滤，拒答边界是否合理
- 决策策略审计日志是否完整记录

**⑤ 端到端执行稳定性**
- ReAct、多 Agent 复杂工作流的端到端路径测试
- 验证多轮交互中状态保持和上下文切换的正确性

### 6. 高并发性能测试中，P99 延迟的 SLA 标准是多少？你如何设计压测场景？

**参考答案：**

**SLA 标准（参考值）：**
- RAG 检索 P99 延迟：≤ 500ms
- 并发召回场景下 P99 延迟：≤ 1000ms
- 具体阈值根据业务场景和用户期望设定

**压测场景设计：**

**① 基准测试**
- 单用户、单接口的响应时间基准
- 逐步增加并发，找出性能拐点

**② 负载测试**
- 模拟真实用户分布的并发量（如 100 并发、500 并发）
- 持续压测 30-60 分钟，观察性能曲线

**③ 峰值测试**
- 突发流量场景，验证系统在超负荷下的表现
- 多路并发召回，验证 P99 是否在 SLA 内

**④ 稳定性测试**
- K8s 部署环境下的集群故障恢复测试
- 验证节点下线、扩容缩容对服务的影响

**工具使用：** JMeter / Locust，结合 K8s Metrics Server 监控资源使用情况。

### 7. 请举例说明如何构建复杂场景下的测试数据？

**参考答案：**

**构建方法：**

   - 直接对测试环境数据库执行 SQL，批量插入符合业务规则的测试数据
   - 使用存储过程生成关联数据（主表+明细表）

   - 从生产环境拉取脱敏数据，在测试环境还原业务场景
   - 确保脱敏规则（去标识化、泛化）符合数据安全要求

### 8. 基于 Dolphin 轨迹数据的可观测测试框架是如何搭建的？如何验证推理过程可追溯性？

**参考答案：**

**框架搭建：**
1. **轨迹数据采集**：通过 Dolphin Runtime 在 Agent 执行过程中自动记录完整轨迹
2. **轨迹解析**：将轨迹数据解析为结构化日志（工具调用、参数、返回结果、执行时长）
3. **断言设计**：基于轨迹设计可观测性断言，如工具调用顺序、参数正确性、执行链路完整性
4. **可视化展示**：将轨迹数据通过链路图形式展示，辅助问题定位

**推理可追溯性验证：**
- **链路完整性**：验证每个用户 Query 对应的完整执行链路是否存在断点
- **决策依据关联**：验证 Agent 输出结果能否回溯到对应的检索文档或工具调用结果
- **异常根因定位**：通过轨迹时间线分析，定位决策异常发生在哪个环节（理解偏差/工具选择错误/执行失败）

**案例：** 在 KWeaver 和 AnyShare 智能助手项目中，通过 Dolphin 轨迹数据验证推理过程可追溯性，实现多起异常问题的快速根因定位。

### 16. 大模型 Prompt 优化和 AI 测试用例生成的实践心得？

**Prompt 优化经验：**

1. **结构化 Prompt 设计**
   - 明确任务目标、输入格式、输出格式

3. **业务适配**
   - 结合业务领域的专业术语，提升 Prompt 的领域适配性
   - 针对不同测试场景（功能测试、安全测试、性能测试）设计专项 Prompt

**AI 测试用例生成实践：**
