# 知识助手RAG 技术原理


## 一、开场：一句话定义 RAG

> "RAG 的全称是 Retrieval-Augmented Generation，检索增强生成。核心思想是：不让大模型凭空编造答案，而是先从企业私域知识库里检索相关片段，再把检索结果作为上下文交给大模型生成答案。这样做有两个核心好处——一是答案可溯源、有据可查；二是企业私有知识能实时接入，不用等模型重新训练。"

---

## 二、整体架构（先说框架，再说细节）

### 两阶段 RAG（Retriever + Reranker）

> "AnyShare 采用的是两阶段 RAG 架构，比单阶段更准。整体分成四个模块——"

| 模块 | 职责 | 面试可以展开讲 |
|---|---|---|
| **Indexing** | 对文档做解析、切片、向量化，建立检索索引 | 多模态解析 → 智能切片 → 向量存储 |
| **Retriever** | 基于关键词/语义/混合策略召回候选内容 | keyword / vector / hybrid 三种召回 |
| **Reranker** | 对召回结果做相关性排序 | 重排为什么能提升答案质量 |
| **Generator** | 结合上下文生成最终答案 | Prompt 构建技巧 |

## 三、文档知识化流程（核心！）

这是白皮书最厚的部分，面试能讲出层次感。

### 第一步：文档解析（多模态）

> "上传的文档可能是 Word、PDF、图片甚至视频。系统先通过 Apache Tika 提取纯文本，再通过 OCR 把图片/扫描件里的文字识别出来。针对复杂版式文档，用 Python 算法库识别章节、标题、表格等逻辑结构，生成一棵内容结构树。"

**关键技术点：**
- 主文档-副文档机制：同一文档解析一次，副文档缓存复用，避免重复计算
- OCR 引擎支持 RapidOCR（CPU/GPU）和第四范式 OCR，按需接入

### 第二步：文档切片（Chunking）

> "不是把整篇文档作为一个整体向量存储，而是切成语义连贯的小片段。切片太大，检索精度低；太小，会丢失关键上下文。AnyShare 采用基于文档版面分析的智能切片，识别段落边界，保持语义完整性。"

**切片策略选择依据：**
- 有明确层级结构 → 按标题/段落切
- 无结构文档 → 按语义相似度动态切 or 固定长度切

### 第三步：向量化（Embedding）

> "每个切片通过嵌入模型转换成高维向量，映射到语义空间里。用户提问时同样向量化，通过余弦相似度在向量数据库里找到最相似的切片。"

**向量检索 vs 关键词检索的本质区别：**
- 关键词检索（BM25）：依赖字面匹配，"AI" 和 "Artificial Intelligence" 查不到
- 向量检索：通过语义理解，表述不同但意思相近的也能找到

### 第四步：知识图谱构建

> "除了向量数据库，系统还基于文档内容构建知识图谱。实体比如'项目文档'、'部门文件夹'，关系比如'隶属于'、'引用了'。图数据库存结构化关系，向量数据库存语义片段，两者协同工作——用户提问时，先通过图谱定位核心实体和关系路径，再通过向量库检索相关文档片段。"

**图谱召回的作用：**
- 支撑复杂关联推理（多跳查询）
- 弥补纯向量检索在关系推理上的不足
- 缓解大模型幻觉——事实依据可验证

## 四、召回策略（面试高频追问）

### 四种召回方式

| 召回方式 | 原理 | 适用场景 |
|---|---|---|
| **关键字召回**（Lexical） | BM25 关键词匹配 | 精确术语查询 |
| **向量召回**（Vector） | 余弦相似度语义匹配 | 语义理解查询 |
| **混合检索**（Hybrid） | 关键词 + 向量融合 | 兼顾精确和语义 |
| **图谱召回**（Graph） | 三元组关系路径查询 | 多跳推理查询 |

> "单独用哪一种都有局限，混合检索是实际生产环境中最常用的方案。"

### 重排序（Rerank）

> "召回回来的候选文档通常很多，直接全丢给大模型成本高且容易稀释答案质量。重排模型（如 Cross-Encoder）对 Top-K 候选再做一次精细排序，只把最相关的几条送去生成。"

## 五、智能体（Agent）体系

### 核心概念区分（面试必问）

> "白皮书里其实有三层概念容易混淆，我梳理了一下——"

| 概念 | 定义 | 层级关系 |
|---|---|---|
| **LLM** | 大语言模型，生成答案的基础 | 底层能力 |
| **Agent** | 能自主决策的智能实体，分主控 Agent 和技能 Agent | 中层调度 |
| **Tool** | 被动工具，被 Agent 调用，本身不决策 | 能力单元 |
| **Data Agent** | 面向具体业务场景创建的智能体，综合调度 | 用户可见层 |

### Agent 工作流程

```
用户提问 → 意图识别 → 数据召回 → 主控 Agent 整合 → LLM 生成 → 结果高亮引用 → 流式展示
```

## 六、技术栈（面试加分项）

| 层级 | 技术 | 说明 |
|---|---|---|
| 前端 | React/Vue + JavaScript | 交互界面 |
| 后端 | Go + 微服务 | 高并发 API 服务 |
| AI 算法层 | Python | 模型推理、解析算法 |
| 编排语言 | Dolphin-lang | 自研流程编排，融合数据处理+变量系统+AI调用+工具集成 |
| 搜索引擎 | OpenSearch | 元数据/文本/向量统一存储 |
| 图数据库 | — | 知识图谱存储 |

> "Dolphin-lang 这个自研编排语言是亮点，它把数据处理、变量控制、模型调用、工具集成都统一了，类似于一个专给 Agent 设计的 DSL。"

## 七、最佳实践场景

### 场景一：FAQ 匹配问答
> "用户问常见问题，系统先通过关键字+向量混合召回最相似的 FAQ，返回原文答案，适合制度类、流程类问答。"

### 场景二：文档检索问答
> "基于私域文档库，用户用自然语言提问，系统检索相关文档片段作为上下文，大模型生成答案并标注引用来源，适合知识库类问答。"

### 场景三：图谱增强问答（Graph RAG）
> "从文档自动构建知识图谱，召回时先查图谱定位实体关系路径，再查向量库补充语义片段，两者融合作为上下文，生成质量更高，尤其适合需要多跳推理的问题。"

---

## 八、面试总结话术

> "这个项目理解最深的是 RAG的端到端链路：从文档上传开始，多模态解析提取内容，智能切片控制粒度，向量化建立语义索引，同时构建知识图谱做关系推理；问答时支持四种召回策略混合使用，通过重排提升相关性，最后把上下文交给大模型生成并标注引用来源。
>
> 相比单纯调 API 的 RAG Demo，这个系统在**工程层面**的核心难点是：多格式文档的解析兼容、切片粒度的平衡选择、召回与重排的协同优化，以及图谱与向量的双路融合。这些是我在项目里真正解决过的问题。"
