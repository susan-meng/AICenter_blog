# Agent 评估体系 · 面试 Q&A

---

## Q1：为什么评估一个 Agent 比传统软件测试难得多？

**✅ 回答：**

传统测试的前提是"给定输入，存在唯一正确输出"，但 Agent 带来了三个不同寻常的挑战：

| 对比维度 | 传统测试 | Agent |
|---------|---------|-------|
| **输出确定性** | 强，确定输出 | 弱，LLM 本质是概率模型 |
| **评估标准** | 客观明确 | 主观模糊（如"什么叫一封好邮件"） |
| **自动化难度** | 低 | 高 |
| **核心目标** | 证明软件没有问题 | **建立一个足够灵敏的感知系统，能在性能退化时尽早发现** |

> 💡 **关键思维转变：从"证明正确"到"感知退化"。Agent 评估的目标不是追求完美，而是第一时间感知到它变差了。

---

## Q2：主流学术评估基准有哪些？各有什么特点？

| 基准 | 设计思路 | 适用阶段 |
|------|---------|---------|
| **BFCL**（Berkeley 函数调用排行榜） | 专门测试模型调用工具的能力，设计务实 | 模型选型 |
| **GAIA** | 466 个真实世界问题，分三个难度级别，评分贴近实际 | 模型选型 |
| **AgentBench**（清华） | 不只测语言，还测在操作系统、数据库等实际环境中完成任务的能力 | 模型选型 |

> ⚠️ 这些是通用基准，要保障自己业务 Agent 的质量，**必须建立专属评估体系**。

---

## Q3：如何构建自己业务的评估体系？

**四步走：**

1. **构建测试集**：50~100 个用例，小而精，覆盖主要场景
2. **采用 LLM Judge**：用强模型评估输出，平衡成本与精度
3. **胜率对比**：有改动时，对比新旧版本，判断新版本是否更好
4. **保留人工验证**：关键场景必须有人工审核这道安全阀

---

## Q4：Agent 评估的核心指标有哪些？

| 指标 | 看什么 |
|------|--------|
| **准确率** | 答案对不对 |
| **工具调用成功率** | Agent 使用工具的能力 |
| **任务完成率** | 最贴近用户实际感受，但自动化评估难 |
| **平均步骤数** | 效率指标，步骤越少 = 执行越快、成本越低 |

---

## Q5：什么是 LLM Judge？为什么有效？

**✅ 回答：**

**LLM Judge** = 用一个大模型（如 GPT-4）去评估另一个模型的输出。

**为什么有效？**
- GPT-4 这类强模型在评估文本质量时，判断力**接近人类专家**
- 能理解问题细微之处、识别事实错误
- 成本远低于大规模人工审核

**通常从三个维度打分：**
- 正确性（答案是否正确）
- 完整性（回答是否完整）
- 相关性（是否回答了用户问题）

---

## Q6：如何深入理解"工具调用成功率"这个指标？

> 工具调用成功率由三个子项共同构成，以天气查询 Agent 为例：

| 子项 | 场景 | 正确行为 |
|------|------|---------|
| **该调时调** | 用户问"北京今天天气怎么样" | Agent 必须调用 `get_weather` 工具 |
| **不该调时不调** | 用户只是说"你好" | Agent 不应调用任何工具，直接回复问候 |
| **参数正确** | 调用 `get_weather` 时 | 参数"北京"必须正确 |

## Q7：Agent 评估如何工程化落地？融入 CI/CD 流程

**✅ 回答：**

理想流程：
1. **触发时机**：开发者修改了影响 Agent 的部件（Prompt、工具描述、模型版本等）并提交代码后
2. **自动触发**：CI/CD 流水线自动触发评估任务
3. **运行测试**：系统运行测试集，用 LLM Judge 打分，计算核心指标
4. **安全阈值**：如果指标相比基线下降超过阈值（如准确率跌 5%），**自动阻止代码合并并通知负责人**
5. **生产监控**：每天抽样真实对话进行评估，发现异常波动立即告警

---

## Q8：完整的 Agent 评估体系总结

**心态转变：**
> 目标不是追求一个完美的系统，而是建立一个足够灵敏的感知体系，当 Agent 变差时能第一时间知道。

**方法结合：**
> 参考 BFCL、GAIA 等学术基准思路 + 构建自定义评估流程（测试集 + LLM Judge + 人工验证）

**工程化落地：**
> 用准确率、工具调用成功率等指标量化表现 + 评估自动化集成到 CI/CD 流水线

---

## 一句话总结

```
评估目标：不是证明正确，而是感知退化
评估方法：学术基准参考 + 业务自定义流程（测试集 + LLM Judge + 人工兜底）
核心指标：准确率 + 工具调用成功率 + 任务完成率 + 平均步骤数
工程落地：CI/CD 自动触发 + 安全阈值阻断 + 线上每日抽样监控
没有评估的优化是盲目的，评估是工程化 Agent 不可或缺的环节
```
