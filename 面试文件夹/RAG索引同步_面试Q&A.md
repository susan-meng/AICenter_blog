# RAG 索引同步 — 面试问答

---

**Q：知识库更新后，RAG 索引如何同步？只说"重新跑一遍向量化"为什么不行？**

**A：**

全量重建成本高、窗口长，在生产环境不可接受。真实业务中文档每天都在新增、修改、删除、权限调整、版本切换，索引必须精细化治理，而不是简单推倒重来。

我的方案是五个维度：

---

## 一、增量更新

不是每次变化都全量重建。给文档和 Chunk 建唯一 ID，记录内容哈希（Content-Hash），**只有内容变化的 Chunk 才重新切片和向量化**。

- 文档 ID + Chunk ID 做主键
- 内容 Hash 做变更判断依据
- 未变化的部分直接跳过，大幅降低成本，避免更新窗口过长

---

## 二、版本管理

同一份文档可能有 V1、V2、V3 多个版本，用户提问时该查哪个版本，系统必须清楚。

MetaData 中必须记录：

- 版本号
- 生效时间
- 发布状态（草稿 / 已发布 / 已废弃）

否则旧版本片段可能和新版本片段一起被召回，答非所问。

---

## 三、删除与失效

很多 RAG 系统只做新增，不会删除。文档从知识库删了，向量还在，模型就会继续引用过期甚至敏感信息。

生产级方案：

- **软删除**：打上失效标记，检索时过滤
- **硬删除**：从向量库中真正删除
- **定期清理**：配合任务定期清理孤立 Chunk

---

## 四、权限同步

文档内容没变，但用户权限变了，RAG 的可见范围必须跟着变。

- 权限 MetaData **不能静态写死**
- 最好能和权限系统联动
- 或者**在检索时实时过滤**，基于当前用户的权限范围裁剪结果

---

## 五、缓存刷新

RAG 系统通常有多级缓存：检索缓存、重排缓存、答案缓存。文档更新后，相关缓存也要同步失效。

否则索引更新了，用户看到的还是旧答案。

缓存失效的绑定维度：

- 文档 ID
- 版本号
- 更新时间

三者任一变化，缓存即失效。

---

## 总结

全量重建是 Demo 阶段的做法。生产级索引同步需要：

1. **基于文档 ID + 内容 Hash 做增量更新**
2. **MetaData 记录版本、时效、权限、状态**
3. **删除时同步失效 / 清理旧 Chunk**
4. **权限变化在检索时实时过滤**
5. **缓存与文档 ID / 版本 / 更新时间绑定刷新**

这就是 RAG 从 Demo 到生产必须补上的**索引治理**。
