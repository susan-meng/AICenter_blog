# Tool 与 Skill 的区别 - 核心 QA 整理

---

## 核心观点

| | Tool | Skill |
|--|------|-------|
| **本质** | 让模型**能动手**的能力 | 让模型**知道该怎么动手**的经验 |
| **解决什么** | 模型本身做不到的事 | 模型能做但不知道按什么流程做 |
| **类比** | 模型伸出去的手 | 手该怎么干活的肌肉记忆 |

**本质区别：**
- **Tool** = 执行，是动作，负责"能不能做到"
- **Skill** = 经验，是规范，负责"做的对不对"

---

## Q1：这题面试官到底在考什么？

**答：** 不是考背概念，而是在考你脑子里有没有一张 **Agent 分层架构图**。

一个真实的 Agent 能干活靠两件事：
1. **能不能真的执行动作**
2. **知不知道遇到某类任务该怎么做**

面试官想看你能不能把这两件事**拆开讲**，讲明白各自解决哪类问题。

如果你只能讲一个维度 → 说明你只接触过其中一半，直接暴露你只看过文档，没落过地。

---

## Q2：Tool 的工程本质是什么？

**答：** Tool 是模型**调不动的真实动作**。

**工程实现：**
- 通过函数调用（Function Calling）或 MCP 协议接进来
- 模型看到工具的描述（名字、参数、用途），决定调哪个、填什么参数
- 系统真正执行，把结果返回给模型

**关键特点：**
- **有副作用**——真正改变世界状态，调一次接口就是真发了一次请求
- **返回结果**——模型拿到结果继续往下推理

**核心：** 执行、动作、模型伸出去的那只手

---

## Q3：Skill 的工程本质是什么？

**答：** Skill **不是执行动作**，本质是一包**知识和流程**。

**结构：** 一个文件夹，包含说明文档，告诉模型遇到这类任务该怎么一步步做，还可以带上模板、脚本、参考资料。

**加载方式——渐进式披露：**
- 平时只有 Skill 名字 + 一句话描述常驻上下文
- 模型一看任务相关，才把完整内容读进来
- 不用就不占位置

**为什么这样设计？**
一个 GitHub MCP 工具集可能暴露 90+ 工具，光描述塞进上下文就要 5 万多 Token。Skill 的按需加载把这个成本压下来。

**核心：** 经验、规范、让模型在动手之前先知道该怎么动

---

## Q4：Tool 和 Skill 在真实 Agent 里是什么关系？

**答：** 不是替代，是**配合**。

| 组件 | 职责 |
|------|------|
| **MCP / Tool** | 给 Agent 接上能力，让他能查、能调、能执行 |
| **Skill** | 教 Agent 把这些能力用对 |

**举例：** 你给模型接了一堆数据接口的 Tool，但他不知道你们公司对账要先校验哪些字段、按什么顺序拉数据、出了异常怎么兜底——这套流程就写进 Skill 里。

**类比：**
- Tool = 手（能执行动作）
- Skill = 肌肉记忆（手该怎么干活的规范）

---

## Q5：线上常见坑是什么？

**答：** 很多人把所有规范都硬塞进系统提示词，结果上下文越堆越长，模型反而抓不住重点。

**Skill 的按需加载就是来解决这个问题的：**
- 该用的时候才进来
- 不用就不占位置

---

## Q6：面试标准回答模板

> **第一句（定维度）：**
> Tool 和 Skill 不是一类东西，一个解决能不能做到，一个解决知不知道怎么做。

> **第二句（讲 Tool）：**
> Tool 是模型调不动的真实动作，靠函数调用或 MCP 协议接进来，有副作用，会返回执行结果。

> **第三句（讲 Skill）：**
> Skill 是打包好的领域经验和流程，是一个带文档和资源的文件夹，靠渐进式披露按需加载，省上下文。

> **第四句（讲关系）：**
> 他俩在真实 Agent 里是配合的，Tool 给能力，Skill 教用法，绝不是谁取代谁。

---

## 一句话总结

> **Tool 是能动手，Skill 是会动手。一个负责执行，一个负责经验。合在一起，才是一个能在真实业务里干活的 Agent。**
