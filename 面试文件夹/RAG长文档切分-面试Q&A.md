# RAG 长文档切分 · 面试 Q&A

---

## Q1：为什么"按长度一刀切"是错误的？

**✅ 回答：**

> 面试官问这个，就是想看你有没有真正落地过 RAG 项目。

**问题在哪**：

计算机只会数字符数，根本不知道语义边界在哪。

```
无脑切：分布式一致性协议
       ↑ 这一刀刚好切在中间

结果：
- 第一个切片结尾：分布式1
- 第二个切片开头：一致性协议

Embedding 模型依赖上下文理解语义，
单独看"分布式1"或"一致性协议"都算不出正确向量。
→ 关键信息根本召回不到 → 模型开始胡编 → 产生幻觉
```

> 💡 **核心问题**：把一句话、一张表、一个完整逻辑从中间砍断，召回的片段缺头少尾，模型自然答错。

---

## Q2：怎么解决这个问题？

**✅ 回答：**

**Overlap（重叠切分）**——让后一个切片的开头和前一个切片的结尾保持一点连接。

```
无 Overlap（灾难）：
切片1：...分布式1        ← 语义断裂
切片2：      一致性协议...

加 20% Overlap（正确）：
切片1：...分布式一致性
切片2：      一致性协议...  ← 完整包含"分布式一致性协议"8个字
```

**核心思想**：用一点点额外存储空间当安全垫，换取语义完整性。

---

## Q3：Overlap 设置多少合适？可以无限加大吗？

**✅ 回答：**

**常见配置**：
- 重叠率：10%~20%
- 切片大小：400~800 token
- 保留 20% 左右的重叠

**Overlap 不是越高越好**：
- 设置过高（如 50%）会导致：
  - **上下文重复冗余**：同一个信息在多个切片里重复出现
  - **检索结果重复**：TopK 里可能召回来自同一个文档的多个重叠片段
  - **存储浪费**：Overlap 部分重复存储，浪费向量库空间
  - **噪声引入**：过多的上下文可能带入无关信息，拉低精确率

> 💡 **本质是平衡**：Overlap 解决语义断裂，但引入过多重复会增加噪声。

---

## Q4：除了 Overlap，还有哪些切片策略？

**✅ 回答：**

| 策略 | 说明 |
|------|------|
| **递归切片（Recursive Splitting）** | 按标题层级、段落自然边界递归切，比固定长度更智能 |
| **语义切片（Semantic Splitting）** | 用模型判断句子边界，按语义连贯性切，成本高但质量最好 |
| **结构感知切片** | 识别标题、表格、代码块，表格和代码块整块保留不拆散 |
| **整块保留切片** | 大段连续内容（如合同条款）尽量不切，保持完整性 |

---

## Q5：面试加分话术（直接背）

> "长文档切分最大的坑就是无脑按字数切，会把语义完整的一句话拦腰砍断，导致向量表征不准确、关键信息召回失败。工业界常用 Overlap 重叠切分，一般 10%~20%，让相邻切片有语义连接。更高阶的会用递归切片或语义切片，顺着标题层级和段落边界来切，表格和代码块尽量整块保留。Overlap 不是越高越好，过高会引入重复噪声、浪费存储。"

---

## 一句话总结

```
一刀切的问题：语义断裂、向量化不准、召回失败
解决方案：Overlap 重叠切分 10%~20%，相邻切片首尾有连接
更优策略：递归切片 + 结构感知切片 + 语义切片
核心原则：语义完整性比存储空间更重要，但 Overlap 过高也会引入噪声
```
