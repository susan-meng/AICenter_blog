# 高并发场景下召回与精排性能测试 · 面试 Q&A

---

## Q1：高并发召回与精排性能测试的核心目标是什么？

**✅ 回答：**

三个核心目标：

| 目标 | 含义 |
|------|------|
| **Latency** | 单次请求的 P50 / P95 / P99 响应时间是否达标 |
| **Throughput** | 系统在给定硬件下能承载的最大 QPS 是多少 |
| **稳定性** | 长时间高并发下，指标是否稳定（内存不泄漏、延迟不漂移） |

> 💡 **关键指标**：P99 latency 是线上 SLO 的核心指标，不能只看平均值。

---

## Q2：测试方案怎么设计？

**✅ 回答：**

### 第一步：明确测试场景

| 场景 | 说明 |
|------|------|
| **单次召回延迟测试** | 固定并发下测 RT 分布 |
| **容量测试** | 逐步加压，找到最大 throughput |
| **长稳测试** | 固定高并发持续运行 30min~2h，观察指标漂移 |
| **突发测试** | 模拟流量突增（AIGC 场景常见），看系统能否扛住 |

### 第二步：构造测试数据

```
向量库规模：100万 / 1000万 / 1亿（按生产规模比例缩放）
召回 TopK：K=10, 50, 100（按业务场景选）
向量维度：768 / 1536（按 Embedding 模型选）
```

### 第三步：并发梯度测试

```
1 → 10 → 50 → 100 → 200 → 500 → 1000 并发
每档记录：QPS、RT(P50/P95/P99)、错误率
找到拐点：延迟开始明显上升的临界点
```

## Q3：召回阶段和精排阶段的性能测试有什么区别？

**✅ 回答：**

| 阶段 | 侧重点 | 常见瓶颈 |
|------|--------|---------|
| **召回阶段** | 大量向量做近似最近邻（ANN）搜索 | 索引结构（HNSW/IVF）、内存带宽、批量查询优化 |
| **精排阶段** | 少量候选用复杂模型重新打分 | 模型推理延迟（embedding / transformer）、GPU 利用率 |

### 召回性能测试关键点：
- 不同索引类型的 QPS 对比（HNSW vs IVF vs PQ）
- 召回数量 K 对延迟的影响（K 越大越慢）
- 批量查询的吞吐提升（单次查 100 vs 100次单独查）

### 精排性能测试关键点：
- 精排候选数量（TopK 召回多少就精排多少）对延迟的影响
- 模型推理是 CPU 还是 GPU，GPU 的话测显存瓶颈
- **打分开销**：精排模型一次 forward 的耗时 vs 召回耗时

## Q4：实际测试中常用的工具是什么？

**✅ 回答：**

| 工具 | 用途 |
|------|------|
| **wrk / locust** | HTTP 接口压测，支持梯度加压 |
| **pytest-benchmark** | 单次 API 延迟精准测量 |
| **vector-db-benchmark** | Faiss / Milvus / Qdrant 等向量库对比 |
| **JMeter** | 企业级复杂压测场景 |
| **Prometheus + Grafana** | 压测过程中实时监控 CPU / 内存 / GPU 利用率 |

## Q5：高并发下精排的性能瓶颈在哪？

**✅ 回答：**

| 瓶颈 | 说明 |
|------|------|
| **精排候选过多** | 召回 TopK=100，但精排模型推理慢，100条全推理扛不住 → 需要**粗精排两层** |
| **GPU 利用率低** | 小 batch 推理效率低，GPU 空转等待数据传输 |
| **精排模型延迟分布不均** | transformer 类模型对输入长度敏感，长文本推理耗时差异大 |
| **打分是串行的** | 有些精排不支持 batch 并行 |

**解法**：
- 粗精排两层：先用一个轻量模型过滤到 10~20 条，再用重模型精排
- Batch 推理优化：攒够一定量再推理，提高 GPU 利用率
- 异步打分：召回结果先返回，精排异步进行（牺牲实时性换吞吐）

## Q6：怎么判断系统是否满足生产 SLO？

**✅ 回答：**

| 指标 | 生产 SLO 参考 |
|------|------------|
| **P99 召回延迟** | < 50ms（ANN 索引）或 < 200ms（复杂查询） |
| **P99 精排延迟** | < 100ms（轻量模型）或 < 500ms（复杂模型） |
| **端到端 P99** | < 300ms（召回+精排+组装上下文） |
| **错误率** | < 0.1%（超时 / 服务端错误） |
| **QPS** | 满足峰值流量的 2~3 倍 |

> 💡 **压测时要把召回延迟和精排延迟分开统计**，不能混在一起算。

---

## Q7：压测过程中需要监控哪些系统指标？

**✅ 回答：**

| 监控维度 | 具体指标 |
|---------|---------|
| **CPU** | 各核利用率、上下文切换频率 |
| **内存** | RSS、GC 频率、内存是否持续增长（泄漏） |
| **GPU** | 利用率、显存占用、温度（长时间压测） |
| **网络** | 向量库 / 精排服务之间的延迟和带宽 |
| **向量库** | 索引内存占用、查询队列长度 |

---

## Q8：面试加分话术（直接背）

> "高并发召回和精排性能测试，核心是把**延迟拆开看**——召回是 ANN 搜索的延迟，精排是模型推理的延迟，两个阶段的瓶颈完全不同。召回看的是索引结构和批量查询优化，精排看的是 GPU 利用率和 batch 推理效率。生产 SLO 我一般按 P99 算，召回 < 50ms、精排 < 100ms、端到端 < 300ms，同时压测时必须监控 GPU 利用率和内存泄漏，长稳测试是最容易发现问题的环节。"

---

## 一句话总结

```
压测目标：找拐点（延迟开始明显上升的临界QPS）
测试设计：梯度加压 + 召回/精排分开统计 + 长稳测试
召回瓶颈：ANN索引结构（HNSW/IVF）、K大小、批量优化
精排瓶颈：GPU利用率、batch大小、粗精排两层架构
SLO标准：P99延迟 + 错误率 + QPS峰值承载力
监控：CPU/GPU/内存/网络/索引队列长度
```
